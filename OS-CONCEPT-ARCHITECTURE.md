# SilicoEco OS 概念架构 v0.1

> 本文只定义产品思想、世界观、核心对象和运行规律，不绑定编程语言、客户端框架、数据库、模型厂商或具体协议。

## 1. 我们究竟要创造什么

SilicoEco 不是一个新的聊天机器人，不是一个新的 Agent 框架，也不是给现有 IM 加上 AI。

它是一套面向人类与数字智能体共同工作的 **Work OS / Organization OS**：

- 它接住一个人或组织的意图；
- 把意图变成可管理的工作；
- 为工作找到合适的人、Agent、工具和知识；
- 让工作在桌面、浏览器、终端、应用和通讯渠道之间连续流动；
- 在关键节点让人做判断、授权和验收；
- 把过程沉淀为证据、产物和组织知识；
- 让下一次工作不再从零开始。

操作系统管理的是计算资源、进程、文件和权限；SilicoEco 管理的是：

> 意图、任务、角色、上下文、能力、决策、产物、权限和组织知识。

因此它更像“工作的操作系统”，而不是“Agent 的应用商店”。

## 2. 从现有项目提取出的四种核心思想

我们不合并现有代码，而是吸收它们各自最有价值的思想。

### 2.1 AnyClaw：智能能力应当随处可运行

核心思想不是 Python、CLI 或某个 Agent Loop，而是：

- Agent 能力不应被某个界面绑死；
- 同一个智能体既可以被人直接使用，也可以被另一个系统调用；
- Skill、工具、记忆和通讯渠道应当可组合；
- 智能体需要长期存在，而不是每次提问都重新出生；
- 本地、后台、终端、服务和远端只是不同运行形态。

SilicoEco 吸收它的“无处不在的智能执行层”思想。

### 2.2 SlockAI：人和 Agent 应在同一个工作空间协作

核心思想不是三栏 UI 或桌面框架，而是：

- Agent 不是藏在按钮后的功能，而是团队成员；
- 人与 Agent 可以进入同一个 Channel、Thread 和 Project；
- Agent 可以被 @、被分配任务、汇报进度和提交结果；
- 多个 Agent 可以拥有不同身份、职责和上下文；
- 协作过程应该对人透明，而不是黑箱执行。

SilicoEco 吸收它的“人机共事空间”思想。

### 2.3 Edith / Jarvis：知识必须成为组织基础设施

核心思想不是终端 TUI 或 pi SDK，而是：

- Agent 不应每次重新阅读整个世界；
- 知识需要被提取、压缩、路由和按需加载；
- 知识产物应开放、自说明、可被不同 Agent 消费；
- 真实工作会产生新知识，知识又应反哺后续工作；
- 企业知识不能只存在于某一次对话或某个 Agent 的私有记忆中。

SilicoEco 吸收它的“组织知识层与渐进式上下文”思想。

### 2.4 Feature Workflow / FDD：复杂工作必须被结构化

核心思想不是 YAML、Git 分支或某组命令，而是：

- 从用户价值出发定义工作；
- 大目标必须切成独立、可验证的小目标；
- 工作之间存在父子、依赖和阻塞关系；
- 执行前要理解上下文，执行后要验证结果；
- 完成不是 Agent 说“完成了”，而是有验收条件和证据；
- 工作结束后要归档、复盘并更新组织知识。

SilicoEco 吸收它的“任务驱动、证据驱动和持续学习”思想。

## 3. SilicoEco 的世界观

整个 OS 建立在一个基本闭环上：

```text
意图 Intent
  ↓
工作 Task
  ↓
上下文 Context
  ↓
执行者 Actor + 能力 Capability
  ↓
行动 Action
  ↓
证据 Evidence + 产物 Artifact
  ↓
判断 Decision
  ↓
知识 Knowledge
  └──────────────→ 进入下一次工作的 Context
```

这个闭环比 Chat 更重要，也比某个 Agent 更稳定。

Agent 会更换，模型会升级，应用会变化，入口会增加，但这个工作闭环不会改变。

## 4. OS 内核

SilicoEco 的内核由四个不可替代的系统组成。

### 4.1 Work Kernel：工作内核

负责回答：

- 当前要完成什么？
- 工作被拆成了什么？
- 谁负责？
- 依赖什么？
- 现在处于什么状态？
- 怎样才算完成？
- 产生了哪些结果？

它管理 Intent、Task、Run、Step、Dependency、Artifact 和 Evidence。

### 4.2 Context Kernel：上下文内核

负责回答：

- 完成这项工作需要知道什么？
- 哪些知识应该常驻，哪些应该按需加载？
- 哪些信息属于项目，哪些属于组织，哪些属于个人？
- 哪些内容是事实，哪些是推测，哪些已经过期？
- 一次工作产生的新认识应写回哪里？

它管理 Project Context、Knowledge、Memory、History、Source 和 Context Route。

### 4.3 Coordination Kernel：协同内核

负责回答：

- 哪个人或 Agent 最适合执行？
- 一项工作应该单独完成、并行完成还是交接完成？
- 什么时候需要人介入？
- 多个 Agent 如何避免重复、冲突和互相覆盖？
- 工作如何从一个入口流转到另一个入口而不丢状态？

它管理 Actor、Role、Assignment、Delegation、Schedule、Conversation 和 Handoff。

### 4.4 Governance Kernel：治理内核

负责回答：

- 谁能看到什么？
- 谁能调用什么能力？
- 什么行动可以自主完成，什么必须批准？
- 谁对某个 Agent 和结果负责？
- 出错时如何暂停、撤销、回滚和追责？

它管理 Identity、Policy、Permission、Approval、Budget、Risk 和 Audit。

这四个内核构成 SilicoEco 的稳定核心。桌面、Web、CLI、IM、模型和浏览器都是外围环境。

## 5. OS 的基本对象

### 5.1 Space：空间

空间是工作发生的边界，可代表个人、团队、部门、项目或公司。

空间决定：

- 成员与 Agent；
- 可见的知识和能力；
- 默认规则；
- 数据边界；
- 工作和产物归属。

### 5.2 Principal：身份主体

凡是能看、说、决定或行动的对象，都是 Principal：

- 人；
- Agent；
- 外部系统；
- 自动化流程；
- 代表某个组织运行的服务。

身份是权限和责任的基础。Agent 不能只是一个提示词文件，它必须是能被授权、暂停、审计和追责的身份主体。

### 5.3 Role：角色

角色表达“在组织中负责什么”，而不是“使用哪个模型”。

例如：

- 产品经理；
- 开发工程师；
- 代码审查员；
- 客服助理；
- 财务核对员；
- 知识管理员；
- 项目协调员。

人和 Agent 都可以承担角色。角色规定责任、能力要求、可见范围和决策边界。

### 5.4 Agent：数字工作者

一个 Agent 由以下要素组成：

```text
身份 + 角色 + 目标 + 能力 + 知识范围 + 行为规则 + 责任人
```

模型只是 Agent 当前使用的一种认知资源，运行环境只是它当前工作的地点，两者都不等于 Agent 本身。

同一个 Agent 可以升级模型、迁移设备或更换工具，同时保持身份、角色、历史和责任关系连续。

### 5.5 Intent：意图

意图是用户或组织想要发生的改变，例如：

- 修复登录问题；
- 研究竞争产品；
- 整理本周客户反馈；
- 为新员工准备入职资料；
- 检查财务异常；
- 建立项目知识地图。

意图可以模糊，系统的第一项工作是把它转化为可执行、可验收的 Task。

### 5.6 Task：工作单元

Task 是 SilicoEco 最核心的业务对象。

一个 Task 至少回答：

- 为什么做；
- 要得到什么结果；
- 谁负责；
- 需要什么上下文和能力；
- 依赖什么；
- 有什么风险；
- 怎样验收；
- 当前进展如何。

Task 可以由人、Agent、流程或外部系统创建，也可以被拆分、委派、暂停、审查和重新执行。

### 5.7 Run：一次执行

Task 表达“工作是什么”，Run 表达“这一次怎么做”。

一项 Task 可以经历多次 Run：

- 第一次由 Agent A 尝试；
- 第二次换 Agent B；
- 第三次由人接管；
- 或者在不同环境重新执行。

这样，一次失败不会等同于整个工作失败。

### 5.8 Capability：能力

Capability 是 Actor 可以做的动作，例如：

- 阅读项目；
- 搜索知识；
- 修改代码；
- 操作浏览器；
- 使用某个业务应用；
- 发送消息；
- 创建审批；
- 调用其他 Agent。

Skill 是能力的使用方法，Tool 是能力的操作接口，Permission 是能力的使用边界。三者不能混为一谈。

### 5.9 Context：工作上下文

Context 不是整段聊天历史，而是某项工作当前真正需要的信息集合：

- 目标与约束；
- 项目事实；
- 相关知识；
- 过去决策；
- 当前环境；
- 参与者信息；
- 已完成工作与未解决问题。

上下文必须可路由、可裁剪、可追溯和可更新。

### 5.10 Conversation：沟通

Conversation 用来讨论、澄清、反馈和协商。

它可以触发 Task，也可以围绕 Task 展开，但 Conversation 不是 Task。否则聊天结束后，责任、状态、依赖和结果都会消失在消息流里。

### 5.11 Artifact 与 Evidence：产物与证据

Artifact 是工作产生的东西：代码、文档、报告、表格、截图、网页、设计、提交记录。

Evidence 是证明工作达到验收标准的信息：测试结果、对比、引用、操作记录、人工确认。

完成必须基于 Artifact 和 Evidence，而不是基于 Agent 的自我声明。

### 5.12 Knowledge：组织知识

Knowledge 是经过组织、可复用、可追溯的信息。它与个人记忆不同：

- Memory 服务于某个 Actor 的连续性；
- Knowledge 服务于整个空间和未来工作；
- History 保存发生过什么；
- Context 是本次工作从三者中选出的有效切片。

## 6. OS 类比

| 传统操作系统 | SilicoEco Work OS |
|---|---|
| User | Human / Agent Principal |
| Process | Task Run |
| Thread | Step / Agent Session |
| Scheduler | Work Coordinator |
| File System | Knowledge + Artifact Space |
| Driver | Capability Provider / Connector |
| Permission | Policy + Approval |
| System Call | 标准工作动作 |
| Shell | Desktop / Web / CLI / IM |
| Process Monitor | Task/Run Activity |
| Package | Skill / Workflow / Knowledge Pack |
| Kernel Event | Work Event |

这个类比的意义在于：外部 Agent 和应用不需要理解整个 OS，只需要通过稳定的“工作动作”加入生态。

## 7. OS 的三条循环

### 7.1 Work Loop：工作循环

```text
提出意图 → 澄清 → 规划/切分 → 分配 → 执行
→ 汇报进度 → 验证 → 决策 → 完成/继续
```

这是每项工作的主循环。

### 7.2 Governance Loop：治理循环

```text
定义责任 → 授权 → 观察 → 发现风险
→ 审批/限制/暂停 → 审计 → 调整权限
```

这是企业可以放心使用 Agent 的前提。

### 7.3 Learning Loop：学习循环

```text
执行产生经验 → 提取事实与方法 → 更新知识
→ 改进 Skill/Workflow/Agent → 下一次工作表现更好
```

没有学习循环，Agent 平台只是在重复消费模型；拥有学习循环后，它才会形成组织资产。

## 8. Task 是整个生态的共同语言

无论任务来自哪里，都先进入同一个工作世界：

```text
桌面 App ─┐
Web       ─┤
CLI       ─┤
浏览器     ─┼─→ Intent / Task → Work Kernel
IM        ─┤                     ↓
外部 Agent ─┤                统一状态与事件
业务系统   ─┘                     ↓
                    任意入口查看、评论、审批、接管
```

各个入口不拥有自己的任务，只展示或操作 OS 中的同一个 Task。

因此：

- IM 中看到的不是“IM 任务副本”；
- 浏览器中看到的不是“浏览器 Agent 的私有任务”；
- WorkBuddy 获取的不是另一套队列；
- 桌面端不是本地任务的唯一真相源；
- 所有入口都围绕一个 Task ID、一个状态和一条事件链协作。

## 9. Task 的生命历程

### 9.1 从模糊意图到 Ready

系统首先帮助用户回答：

- 真正想改变什么？
- 结果面向谁？
- 完成标准是什么？
- 是否需要拆分？
- 有哪些约束和风险？
- 需要哪些知识和权限？

只有具备足够执行条件后，Task 才进入 Ready。

### 9.2 从 Ready 到执行

协调内核根据以下因素选择 Actor：

- 角色是否匹配；
- 是否具备所需能力；
- 是否能访问必要上下文；
- 当前是否有容量；
- 成本、速度和质量要求；
- 风险等级和治理规则；
- 是否需要多 Agent 或人机协作。

### 9.3 执行过程

执行不是黑箱，至少应持续产生：

- 当前计划；
- 进展事件；
- 关键行动；
- 阻塞与问题；
- 需要的审批；
- 中间产物；
- 成本和时间；
- 对目标或计划的变更。

### 9.4 验证与完成

Task 的完成分为三层：

1. **执行完成**：Actor 已停止工作并提交结果；
2. **验证通过**：结果满足自动或人工验收条件；
3. **业务完成**：负责人接受结果，后续影响已处理。

系统不能把这三者压缩成一个“Done”。

### 9.5 完成后的写回

完成后需要判断：

- 哪些事实应该进入项目知识？
- 哪些方法可以沉淀为 Skill？
- 哪些步骤可以固化为 Workflow？
- Agent 的职责或权限是否需要调整？
- 是否产生新的任务？

## 10. FDD 在 OS 中的位置

FDD 不是 SilicoEco 的全部工作方式，而是 Work Kernel 上的一种“结构化工作制度”。

它提供几个普适原则：

- 以用户价值而不是技术层切分；
- 每个子任务独立产生可验证价值；
- 显式表达依赖；
- 在执行前加载项目上下文；
- 通过证据完成验收；
- 完成后更新知识。

因此 OS 中可以存在多种制度：

- Feature Development；
- Bug Resolution；
- Research；
- Content Production；
- Customer Support；
- Recruiting；
- Financial Review；
- Knowledge Governance。

它们共享 Task、Actor、Context、Evidence 和 Governance 内核，但拥有不同的阶段、角色、模板和验收方法。

## 11. 人与 Agent 的组织模型

### 11.1 Agent 不是工具列表中的一个条目

Agent 在组织中应该拥有：

- 名字和身份；
- 清晰职责；
- 所属团队；
- 人类责任人；
- 能力与知识边界；
- 可见范围；
- 工作容量；
- 行为规则；
- 绩效记录；
- 生命周期状态。

### 11.2 人仍然掌握方向和责任

人类的关键职责不是亲自执行每个步骤，而是：

- 定义目标；
- 设计组织；
- 分配责任；
- 设定边界；
- 在不确定和高风险节点做决定；
- 验收结果；
- 对最终影响负责。

### 11.3 Agent HR 的真正含义

Agent HR 管理的不是“提示词配置”，而是数字工作者的组织生命周期：

```text
定义岗位 → 选择/创建 Agent → 试用 → 评估
→ 入职 → 授权 → 分配工作 → 观察
→ 培训/升级 → 调岗 → 暂停 → 离职与知识交接
```

它需要与安全、项目管理和业务负责人共同治理，而不是拥有无限权限。

### 11.4 混合团队

未来的基本团队单元可能是：

```text
1 位人类负责人
  ├─ 产品/规划 Agent
  ├─ 执行 Agent A
  ├─ 执行 Agent B
  ├─ 审查 Agent
  └─ 知识 Agent
```

也可能是多个人与多个 Agent 共同组成。SilicoEco 管理的是角色、责任和工作关系，而不是预设固定团队形态。

## 12. 上下文与知识体系

### 12.1 上下文不应靠无限堆积

上下文采用渐进结构：

```text
Layer 0：方向与路由
  组织规则、项目地图、角色、关键约束

Layer 1：工作速查
  与当前 Task 直接相关的模块、流程、决策和检查方式

Layer 2：深层事实
  代码、文档、历史记录、详细业务规则和原始来源
```

Actor 先理解地图，再按任务加载局部知识，最后在必要时下钻原始事实。

### 12.2 知识需要治理

知识对象应该具备：

- 来源；
- 所有者；
- 适用范围；
- 更新时间；
- 可信度；
- 与其他知识的关系；
- 是否经过人工确认；
- 是否已经过期或冲突。

### 12.3 工作即知识生产过程

每个 Task 都可能产生四类知识：

- 事实：系统现在是什么样；
- 决策：为什么选择这种做法；
- 方法：以后遇到同类问题怎么做；
- 边界：哪些做法会失败或不能使用。

OS 应在任务结束时引导写回，而不是把所有消息永久保存后称为“记忆”。

## 13. 能力生态

SilicoEco 的生态不围绕某一种插件格式，而围绕六类可交换资产。

### 13.1 Actor Provider

提供一个可承担工作的外部 Agent、模型系统或人类服务。

### 13.2 Capability Provider

提供一种可执行能力，例如浏览器、代码、邮件、设计、财务系统或企业应用。

### 13.3 Workflow Pack

提供一套结构化工作制度：阶段、角色、条件、检查、审批和完成规则。

### 13.4 Skill Pack

提供解决某类问题的方法和操作知识。

### 13.5 Knowledge Pack

提供某个组织、行业、项目或领域的知识结构和内容。

### 13.6 Surface Adapter

让用户从桌面、浏览器、终端、IM 或业务系统进入同一个工作世界。

所有生态资产都必须声明：

- 它是什么；
- 能做什么；
- 需要访问什么；
- 由谁负责；
- 产生什么；
- 有什么风险；
- 如何被观察和停止。

## 14. 与 WorkBuddy 和其他平台的关系

SilicoEco 不要求用户抛弃现有 Agent 平台。

它可以处于三种关系中：

### 14.1 SilicoEco 作为工作源

WorkBuddy 或外部 Agent 从 SilicoEco 获取被授权的 Task，完成后回传状态、证据和产物。

### 14.2 SilicoEco 作为执行者

外部平台把工作交给 SilicoEco，由 SilicoEco 的 Agent、浏览器、应用和工作流完成。

### 14.3 SilicoEco 作为协同空间

来自不同平台的 Agent 以组织成员身份进入同一个 Space，在统一任务和治理规则下协作。

SilicoEco 的壁垒不是“支持多少种调用协议”，而是能够让这些不同来源的智能体共享：

- 一个工作事实；
- 一套责任体系；
- 一个知识环境；
- 一条可审计的过程；
- 一个持续学习闭环。

## 15. 产品空间

用户看到的产品不应按底层技术划分，而应按工作问题划分。

### 15.1 Today / Inbox

回答“现在什么最需要我的注意”：待决定、待审批、被阻塞、失败、即将到期、Agent 请求帮助。

### 15.2 Work

所有目标、任务、依赖、执行过程、证据和产物。

### 15.3 Collaborate

Channel、Thread、评论、@、交接和人与 Agent 的讨论。

### 15.4 Team

人、Agent、角色、团队、容量、职责和工作关系。

### 15.5 Knowledge

项目地图、组织知识、决策、方法、来源、健康度和更新情况。

### 15.6 Capabilities

可以被调用的 Skill、工具、应用、浏览器、外部 Agent 和工作流。

### 15.7 Governance

权限、审批、风险、预算、审计、Agent 生命周期和组织政策。

这些产品空间可以通过桌面、Web、终端和 IM 呈现不同深度，但语义保持一致。

## 16. 桌面、Web、终端、浏览器和 IM 的角色

### 16.1 桌面

个人工作的主控制台，适合观察长任务、本地资源、多个空间和复杂产物。

### 16.2 Web

团队和组织的共享工作面，适合跨设备访问、项目协作和管理治理。

### 16.3 终端

高密度、低干扰的 Agent 工作面，适合开发者、自动化、深度任务和运行状态观察。

### 16.4 浏览器

既是信息来源，也是行动环境。它让 Agent 在真实业务页面中工作，同时让人观察、接管和批准关键动作。

### 16.5 IM

通知、轻量协作和快速决策入口，不应成为复杂工作状态的唯一保存位置。

它们不是五套产品，而是同一个 OS 的五种 Shell。

## 17. 治理原则

### 17.1 能看到不等于能行动

信息读取、生成建议、执行变更、对外提交必须分别授权。

### 17.2 Skill 不等于权限

Agent 知道如何发邮件，不代表它被允许发邮件。

### 17.3 一次批准不等于永久授权

审批应针对具体 Task、具体动作、具体范围和有限时间。

### 17.4 自主程度是连续谱

```text
只读建议
→ 生成草稿
→ 执行前逐步批准
→ 仅高风险动作批准
→ 在明确边界内自主执行
```

不同 Agent、任务和空间可以处于不同自主等级。

### 17.5 责任链不能消失

每个 Agent、Task、决策和外部行动都必须能回答：

- 谁创建的；
- 谁负责；
- 谁授权的；
- 谁执行的；
- 谁验收的。

## 18. SilicoEco 与普通 Agent 平台的差异

| 普通 Agent 平台 | SilicoEco OS |
|---|---|
| 以 Prompt/Chat 为中心 | 以 Intent/Task 为中心 |
| Agent 是功能或机器人 | Agent 是有身份和责任的数字工作者 |
| 关注一次执行 | 关注完整工作生命周期 |
| 每个入口一套状态 | 所有入口共享工作事实 |
| 上下文等于聊天历史 | 上下文是被路由的组织知识切片 |
| Tool 能调用就能使用 | Capability 与 Permission 分离 |
| 完成靠 Agent 声明 | 完成靠验收、证据和责任人 |
| 日志是调试信息 | Event 是协作、审计和学习资产 |
| 多 Agent 等于并行 Prompt | 多 Agent 是角色、依赖、交接与治理 |
| 插件扩展工具 | 生态扩展 Actor、Workflow、Skill、Knowledge 和 Surface |

## 19. 产品建设顺序

### Foundation 1：建立工作世界

先定义并跑通：

- Space；
- Principal；
- Role；
- Intent；
- Task；
- Run；
- Context；
- Artifact；
- Evidence；
- Event；
- Approval。

目标不是做全，而是让这些对象的边界稳定。

### Foundation 2：建立个人闭环

一个人在任何入口提出意图，Agent 可以理解、执行、汇报、请求批准、提交证据并写回知识。

### Foundation 3：建立团队闭环

多个人和多个 Agent 可以围绕同一 Task 协作、交接、并行和审查，任何入口看到相同状态。

### Foundation 4：建立组织闭环

组织能够定义角色、Agent 岗位、权限、自主等级、预算、责任人和审计规则。

### Foundation 5：建立生态闭环

外部 Agent、能力、工作流、知识和通讯入口可以进入统一工作世界，并遵循相同治理规则。

## 20. 第一版 OS 验证标准

第一版的成功不以功能数量衡量，而看是否证明以下十件事：

1. 同一个 Task 可以从桌面、终端和 IM 被连续操作，而不是复制三份；
2. Task 与一次 Agent Run 明确分离；
3. 人和 Agent 都能成为负责人、执行者或审查者；
4. 一个 Agent 可以把工作交给另一个 Agent，并保留责任链；
5. 工作过程能产生可理解的进度，而不是只有 token 流；
6. 高风险行动会在正确时机请求批准；
7. 完成时有产物和证据；
8. 工作产生的新知识能被下一项工作按需加载；
9. 更换模型或运行环境不会丢失 Agent 的组织身份；
10. 外部 Agent 可以进入同一个 Task 世界，而不产生另一套状态。

## 21. 北极星指标

不应以“消息数”“Agent 数”或“token 消耗”作为产品核心指标。

更有意义的指标包括：

- 从意图到可执行任务的时间；
- 无需重新解释上下文的任务比例；
- 有明确责任人和验收条件的任务比例；
- 人工介入发生在正确决策点的比例；
- 一次验收通过率；
- 任务跨入口流转时的状态一致率；
- 组织知识被后续任务复用的比例；
- Agent 产生结果的可追溯率；
- 相同类型任务随时间的成本和质量改善；
- 人从重复执行转向方向判断所节省的时间。

## 22. 最终产品定义

SilicoEco 是：

> 一个让人类、Agent、知识、应用和工作流在统一任务、统一上下文和统一治理下协作的组织操作系统。

它的最小内核是：

```text
Work + Context + Coordination + Governance
```

它的最小闭环是：

```text
Intent → Task → Action → Evidence → Decision → Knowledge
```

它的长期壁垒是：

```text
组织工作图谱 + 组织知识图谱 + Agent 责任与能力图谱
```

桌面 App、Web、终端、浏览器、IM、MCP、CLI、模型和技术栈都可以变化；只要这三个图谱和三个循环持续存在，SilicoEco 就仍然是同一个 OS。
